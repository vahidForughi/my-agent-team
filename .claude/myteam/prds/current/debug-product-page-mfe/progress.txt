# - Always use conventional commits: feat:, fix:, chore:
# - Run quality gates before committing (see config.yaml > defaults.qualityGates)
# - Micro-frontends: run `npm run setup` (build:packages) before serving; remotes serve on 4200-4204.
# - The host loads remotes at runtime via @module-federation/runtime in host/src/microFe/MicroFrontendApp.tsx
#   (registerRemotes + loadRemote('<remote>/ConsoleMicroApp') from {remoteUrl}/remoteEntry.js); failures render ErrorBoundaryFallback.
# - DO NOT call @module-federation/runtime init() from the host: the host is already initialized as a MF host at
#   build time by @nx/module-federation. Re-calling init() (esp. with a different `name`) throws RUNTIME-010
#   ("The name option cannot be changed after initialization"). Use registerRemotes([{name, entry}]) to add remotes
#   to the existing runtime instance, guarded so each remote registers at most once.
# - Debugging MFE load in a headless browser without installing Chromium: Playwright can drive system Edge via
#   chromium.launch({ channel: 'msedge' }); capture page.on('console'|'pageerror'|'requestfailed'|'response').
# - NX version skew in micro-frontends/package.json: nx core + @nx/eslint|jest|js|web|workspace are 21.6.11 but
#   @nx/module-federation|react|webpack|playwright are 22.7.5. The nx daemon may fail to compute the project graph
#   in restricted environments; typecheck a single project directly with `tsc --noEmit -p <proj>/tsconfig.app.json`.

## [2026-06-16 13:50] - US-001
- Reproduced and root-caused the product-page MFE load failure against the running suite
  (host :4200, store :4201, gateway :8010). frontend-developer traced the load path; evidence-collector
  captured live browser proof via Playwright + system Edge (channel 'msedge').
- Evidence:
  - Network: GET http://localhost:4201/remoteEntry.js => 200 text/javascript (~3 MB), a valid `var store`
    webpack MF container that exposes `./ConsoleMicroApp`. The remote IS served correctly (rules out 404/CORS).
  - Console (host): "[MicroFrontendApp] Error loading store: Error: [ Federation Runtime ]: The name option
    cannot be changed after initialization ... #RUNTIME-010" at ModuleFederation.initOptions -> init ->
    loadMicroFrontend. Page rendered the ErrorBoundaryFallback ("Failed to Load Micro Frontend").
- Classification: failing layer (b) — runtime init/share-scope. Root cause is the host calling
  @module-federation/runtime init() with name '@ecommerce-host' on an already-initialized MF host
  (host/src/microFe/MicroFrontendApp.tsx). The remote, exposes, and serve-on-:4201 are all correct.
- Files changed: none (diagnosis only); prd.json US-001 -> passes:true.
- **Learnings for future iterations:**
  - The MF runtime is a singleton; the host is initialized at build time by @nx/module-federation, so a
    second init() with a different name is fatal (RUNTIME-010). Add remotes with registerRemotes instead.
  - A 200 remoteEntry that starts with `var <name>;` is a healthy global container — look upstream at the
    host runtime, not the remote, when loadRemote throws.

## [2026-06-16 14:30] - US-004 (partial — regression guard added & verified; full data E2E blocked)
- Added a data-independent regression guard for the RUNTIME-010 bug in
  micro-frontends/e2e/module-federation.spec.ts: "should mount the Store remote without a RUNTIME-010
  init error". It navigates to /store and asserts (a) the host logs "[MicroFrontendApp] Successfully
  loaded store", (b) no console/page error contains RUNTIME-010 / "name option cannot be changed", and
  (c) the ErrorBoundary text "Failed to Load Micro Frontend" is absent.
- Why a new test: the pre-existing "should handle module loading errors gracefully" only checks
  [role="alert"], .error-boundary — but the fallback is an antd <Result> (.ant-result, no role=alert),
  so that test did NOT catch this regression. The new guard keys off the actual symptoms.
- Verified green against the running host via Edge (chromium not installed locally): created a throwaway
  playwright.edge.config.ts (channel 'msedge', reuseExistingServer), ran `playwright test -c ... --grep
  "RUNTIME-010"` -> 1 passed (6.0s), then deleted the throwaway config. In CI the test runs under the
  standard `chromium` project in playwright.config.ts.
- Files changed: micro-frontends/e2e/module-federation.spec.ts (+ this progress note).
- NOT marking US-004 passes:true: the remaining ACs (product-detail render with data, full `nx run-many
  test`/e2e suite) depend on the Catalog backend (:8000) which is down and can't be started here (no
  Docker), and the nx daemon can't build the project graph in this sandbox.
- **Learnings for future iterations:**
  - When guarding an MFE-load regression, assert the positive success log + absence of the antd Result
    title text; do not rely on role="alert"/.error-boundary (antd <Result> sets neither).
  - To run Playwright without bundled Chromium, drive system Edge via channel 'msedge' + reuseExistingServer.
